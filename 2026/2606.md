## 背景

今年国产大模型大爆发。各个模型都号称自己对标Claude Opus 4.5甚至Claude Opus 4.6。

尤其是最近出来的GLM 5.1

![2606-glm5.1.png](2606-glm5.1.png)

还有 Kimi K2.6 。

![2606-kimi-k2.6.jpg](2606-kimi-k2.6.jpg)

是不是看起来挺唬人的。

看起来噱头十足。但是实际上有“跑分没输过，实际体验没赢过”的嫌疑。
所以我这次记录了一下近期一些真实使用场景的体验对比。

同时，各个国产大模型的Coding Plan，定价模糊。为了更容易对比各个厂商提供的订阅额度到底有多少。

这次测试GPT和Claude都是使用VSCode内置的copilot。GLM 5.1和Kimi K2.6都是使用Kilo Code，思考强度GLM 5.1是High，Kimi K2.6是Max。

## 实际体验效果

MiniMax M2.5和千问 3.6 Plus。我之前简单测试过，即便是简单任务，也经常理解错误和中断。和Kimi、GLM差距很遥远（盲猜是参数规模小导致的，再怎么优化也干不过其他几家参数量碾压）。
所以后面就没再关注了。

主要关注Kimi和GLM的模型系列。这两个模型在日常简单任务都表现都挺不错，虽然干不过GPT和Claude，但是性价比高且国内使用无障碍呀。

我这次的真实业务场景体验和一些评测不一样的是，我们不是从零开始创造一个东西。而是要它符合之前的项目规范和设计。
大部分评测从零开始造轮子的，其实偏离大多数人的AI使用场景。大语言模型往往从0到1的体验会远远好于1到100。越是复杂的项目，背景知识就越多，模型就越容易产生幻觉。
而且AI写的代码的维护难度大多数情况下比有一定架构能力的工程师写的要难维护。时常会有大量相似代码，还会实现成大差不差，单元测试通过，但是有那种你不自己分析比较难发现的暗坑。

> 曾经我让Claude Opus 4.6照着C++一个接入层实现来写golang版本，结果它没限制缓冲区长度会溢出。断线重连的token验证单元测试绕过去了，根本不对。
> 并且没有实现恢复容灾恢复逻辑。如果出现故障了就完全GG。所以大家让AI写的代码，除非是小工具或者玩具，务必要Review。
> 否则如果任由它在这类问题上到处埋坑。特别是在服务端研发里，各个环节全靠偶然无故障打通。等到时候哪个环节抖动一下。业务就GG了。

本次测试执行的任务内容大致上如下：

- 编写了一个AI agent指令和skills，告诉AI如何访问阿里云API、如何鉴权、如何查询文档以及文档目录。
- 让大语言模型操作阿里云的SLS服务，执行任务:
  - 分析已有的日志结构和索引。
  - 创建LogStore和定时SQL任务取做数据清洗并入库。
  - 补充遗漏和索引设置。
  - 创建仪表盘，编写查询语句。

### 第一个场景：自动阅读文档并创建阿里云资源

在编写查询语句和访问API中，无论是GPT 5.4(high思考强度)，Claude Opus 4.7，GLM 5.1，Kimi K2.6均无问题。
（只有一个查询语句中Kimi K2.6的分组逻辑错误，应该返回多个分组，但实际返回的是sum）
但是所有的模型（包括PT 5.4(high思考强度)和Claude Opus 4.7）的第一版SQL，在JOIN的表中都缺少数据范围的限制。会导致扫描过大的数据集。
且所有模型都能在一次提示后正确分析并优化正确。

在一个仪表盘中，可能是阿里云的文档本身有误。图表的单位设置不正确，然后这个问题我分别让GPT 5.4(high思考强度)，GLM 5.1和Kimi K2.6去尝试解决。
这个比较就比较明显了。

| 人工交互阶段 | GPT 5.4(high思考强度)                          | GLM 5.1                                                                          | Kimi K2.6                                    |
| ------------ | ---------------------------------------------- | -------------------------------------------------------------------------------- | -------------------------------------------- |
| 问题提出     | 分析出了问题的大概原因，尝试解决但失败         | 分析出了问题的大概原因，尝试解决但失败                                           | 分析出了问题的大概原因，尝试解决但失败       |
| 第一次反馈   | 尝试分析提取UI的可选项，找到正确的值后修复成功 | 分析出了问题的大概原因，发现自己分析失败，提示用户设置一个正确的后自己再对比解决 | 分析出了失败的原因，但是尝试再次解决仍然失败 |
| 第二次反馈   | -                                              | 人工修正一个图表后，能够把其他的有问题图表也修复正确。                           | 分析出了失败的原因，但是尝试再次解决仍然失败 |
| 第三次反馈   | -                                              | -                                                                                | Andante订阅的5小时限频耗尽，失败终止         |

### 第二个场景：修复仪表盘错误

Kimi K2.6创建的图表里的系列标签没有按要求显示出分组信息。

> 这个问题起因是阿里云的文档页面失效了。即便数据正常分组了，图表里显示仍然不正常。AI写的SQL和分组本身是正确的。

- Kimi K2.6
  - 两次反馈错误并尝试修复后仍然没能正确修复。
- GLM 5.1
  - 第一次反馈错误后，先通过API拉取数据，但是拉取的流程有问题导致没拉倒任何数据。然后启动了浏览器来辅助分析。浏览器需要我手动去登入，也不是全自动的。然而，最后还是没解决。
  - 第二次通过截图反馈修复失败。但是Kilo Code没有正确调用GLM提供的视觉识别MCP，没能分析截图。
  - 第三次反馈修复失败后，提示词加入要求阅读文档，并自动修复错误文档链接。执行过程应该是触发了Kilo Code的Auto compact。然后卡死了。
  - **失败结束**
- GPT 5.4(high思考强度)
  - 第一次反馈错误后，先通过API拉取数据分析修复失败。
  - 第二次反馈修复失败后，改为写死的方式修复series。虽然能够显示多条线，但不符合动态拆线的目的。属于有效尝试。
  - 第三次反馈不符合需求后，改回来了。
  - 第四次主动提示尝试换一种图之后，正确分析出了可以使用流图，但是把图改没了（图表类型字段设置错误）。
  - 第五次提示图改没了之后，认为图的类型参数传错，还原回了之前那有问题的版本。
  - **失败结束**
- Claude Opus 4.7
  - 第一次反馈错误后，分析认为数据是正确的。
  - 第二次反馈数据正确，但是绘图显示不对后，AI尝试修改优化，但分析修复失败。
  - 第三次主动提示尝试换一种图之后，选用柱状图（错误结论），分析修复仍然失败。
  - **失败结束**

在没有人工提示的情况下，几个模型都没能修复。
其中GLM 5.1卡死一次（之前也碰到其他情况GLM 5.1卡死一次），不排除是Kilo Code的BUG。
虽然没最终解决问题，但是GPT 5.4在人工提示下的分析过程是正确的，Claude Opus 4.7在第一次就正确识别了数据是争取的。

### 第三个场景：分析Kilo Code的模型调用量

- GLM 5
  - 同时存在老版本的日志和数据记录，一次找到正确的新版数据
- Kimi K2.6
  - 第一次找到老版本数据，分析不出来
  - 第二次提示新版本数据位置和格式可能不一样之后，可以正确找到数据并分析出结果。

> 整个操作过程中Kimi K2.6最快，远远快于其他两个。GLM 5.1最慢。

## Coding Plan额度

虽然各个模型命中缓存和不命中缓存价格相差巨大。但是鬼知道它啥时候命中缓存。所以我这里不去区分这个，就用正常使用的过程来评测。缓存命中率也会和正常使用的场景相近。

### 腾讯云和阿里云 Coding Plan

腾讯云和阿里云 Coding Plan刚出来那会儿，真是性价比很高。真正的主打一个量大管饱。
我一个人用虽然使用频率不高吧，但是基本上就没使用任意一个限额用到 5% 的。所以放大使用倍率没有太大的意义。

### 月之暗面/Kimi

我之前其实没有买Kimi会员，因为它的官方介绍页里对限额的描述模糊不清，总感觉有猫腻。
但是现在各大云厂商的Coding Plan都基本不更新了（都在推更贵得离谱的Token Plan）。
为了体验新模型的效果，所以我从最便宜的订阅开始，一档一档升级（由于贫穷最高档买不起没测试）开看实际它到底有多少用量。

## 总结

总体效果个人感觉还是GLM 5.1强一些。虽然不及GPT和Claude，但是作为补充，去完成一些不是特别深度的编码和长任务都基本OK。
还是怀念两大管饱的Coding Plan。之前感觉Kimi会员很贵，但是实际用下来应该也还好。它额度模糊总给人很不好的感觉。而且后面如果暗改计算方式也不透明。
Kimi会员的价格比GLM的贵50%，但是给的量翻了一倍。而且还有其他的一些附加服务，目前来说应该是更具有性价比一点。

现在各个云厂商的Coding Plan都绝版了，我也在犹豫要不要后面如果不更新的话就退掉算了。
也希望能给其他人提供选择参考。
